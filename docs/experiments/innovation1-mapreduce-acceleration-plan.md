# Innovation 1 Map/Reduce Acceleration Experiment Plan

**Date:** 2026-07-02

**Status:** local diagnostic passed

## Purpose

Evaluate whether Map/Reduce-style parallelism can accelerate Innovation 1
SPN/PRESENT experiments without changing the cipher protocol, labels, negative samples,
validation data, model architecture, metric computation, or checkpoint logic.

This is a speed and infrastructure experiment. It is not cryptanalytic evidence and
must not be reported as a model-quality result.

## Research Question

Can the existing chunked disk dataset cache path speed up PRESENT-80 r7 dataset
generation when the sample space is split into independent chunks and generated by
multiple workers?

## Hypothesis

PRESENT data generation is mostly CPU-bound and chunk-independent. A Map/Reduce-style
cache builder should improve rows/second at medium and large scales:

```text
Map:    worker processes independently generate deterministic positive/negative chunks
Reduce: write generated chunks into disk-backed cache arrays and metadata
Reuse:  later model rows reuse the same parameter-matched cache
```

## Existing Implementation Under Test

The current implementation already contains a conservative Map/Reduce-like cache path:

```text
src/blockcipher_nd/data/cache/disk.py
```

Key behavior:

```text
make_chunked_differential_dataset()
_iter_generated_chunks()
ProcessPoolExecutor(max_workers=workers)
_generate_chunk_worker()
features.npy / labels.npy / metadata.json reduce output
```

This experiment tests that path before introducing a heavier shard-writer or DDP
training implementation.

## Gate

This local diagnostic is successful if:

1. Workers 1, 2, and 4 all generate valid cache artifacts.
2. Each result writes `features.npy`, `labels.npy`, and `metadata.json`.
3. The output metadata records the requested worker count.
4. At least one multi-worker setting improves rows/second over workers=1, or the result
   clearly shows that this local scale is too small to amortize multiprocessing overhead.

Interpretation:

- The local diagnostic passed all cache correctness gates.
- Multi-worker chunk generation improved throughput even at this small local scale.
- This supports the hypothesis that PRESENT dataset generation is a good target for
  Map/Reduce-style acceleration.
- This is still speed evidence only. It does not say anything about model accuracy or
  Innovation 1 route quality.
